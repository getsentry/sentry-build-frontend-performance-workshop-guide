## Problem: Silent CVV Validation Failures

Your e-commerce store recently migrated payment gateways. Everything seems fine, but checkout completion rates have mysteriously dropped by ~25%. No errors are being thrown, but users are silently failing to complete purchases.

### The Mystery

- Checkout button appears to work
- No error messages shown to users
- No JavaScript errors in Sentry
- Payment validation logs show CVV failures, but users can't see why

:::tip[Reproduce the Issue]
Before diving into the code, let's experience this failure from a user's perspective:

1. Open your e-commerce app and add items to your cart
2. Navigate to the checkout page
3. Try to complete a purchase using an **Amex card** with a 4-digit CVV:
   - Card Number: `3782 8224 6310 005`
   - CVV: `1234`
4. Click the checkout button

Notice what happens (or doesn't happen) - no error message appears, but the checkout doesn't complete either. This is the silent failure users are experiencing.

Now open **Session Replay** in Sentry to watch the recording of your checkout attempt and see exactly what the user saw.
:::

### What Went Wrong?

After investigating with **Session Replay** and **Sentry Logs**, you discover:

```tsx
// The broken validation code
const validateCVV = (cvv: string) => {
  if (cvv.length !== 3) {
    // ‚ùå Logs to Sentry but user sees nothing!
    console.error(
      `CVV validation failed: Expected 3 digits, got ${cvv.length}`
    );
    return false;
  }
  return true;
};
```

**The issues:**

- Visa/Mastercard CVVs: 3 digits (123) ‚úÖ Works
- Amex CVVs: 4 digits (1234) ‚ùå Fails silently
- ~25% of US credit cards are Amex - that's 25% of users blocked!

## The Fix: Show Validation Errors & Add Structured Logging

Let's fix this by showing validation errors, supporting both card types, and adding **Sentry Logs** for critical debugging paths.

:::note[Understanding the Logging APIs]

This Cart component uses the **Console Logging Integration**, which automatically captures `console.log()`, `console.warn()`, and `console.error()` calls and sends them to Sentry. Both traditional console methods and the `Sentry.logger` API are available:

**`console.log()` with parameters:**

When you pass multiple parameters to console methods, they're automatically parsed as searchable attributes:

```tsx
console.log('Starting checkout validation for', itemCount, cartTotal);
// Creates: message.parameter.0 = itemCount, message.parameter.1 = cartTotal
```

**`Sentry.logger` API:**

A dedicated logging API with explicit severity levels (`trace`, `debug`, `info`, `warn`, `error`, `fatal`) and context objects:

```tsx
// With context object
Sentry.logger.warn('Checkout blocked: CVV validation failed', {
  cartTotal,
  itemCount,
});
```

**`Sentry.logger.fmt` enhancement:**

Creates template strings where variables become searchable attributes with semantic names:

```tsx
Sentry.logger.error(Sentry.logger.fmt`Checkout error: ${errorMessage}`);
// Creates: message.parameter.0 = errorMessage (with template preserved)
```

All three approaches send logs to Sentry where they're searchable, queryable, and correlated with traces, replays, and errors.

Learn more: [Sentry Logs Documentation](https://docs.sentry.io/platforms/javascript/guides/react/logs/)
:::
